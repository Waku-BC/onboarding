Domain Restriction

Compare the same Brightcove asset through video tag, iframe, and hosted player URL access. This page is designed to show why iframe testing can be misleading when engineers assume the visible page origin is always the evaluated playback context.

Common misconception

Same asset, different execution context. Restriction results change depending on where playback is actually evaluated.

Playback / Security / Embed Context

Wrong assumption

“I restricted playback to my site domain, so the iframe should behave the same as the in-page embed.”

Actual behavior

Restriction should succeed when playback is evaluated against the site domain and fail when the observed context shifts elsewhere.

Understanding player runtime

A simple visual explanation for why engineers often ask whether playback is “really happening on players.brightcove.net.”

Script source ≠ runtime context

Plain-English explanation

What “runtime” means on this page

The key distinction is where the player script is delivered from versus where the player actually executes.

Support-ready

When someone says, “If the player script loads from players.brightcove.net, doesn’t that mean playback always happens there?”, the answer is not always.

The Brightcove Player JavaScript is served from players.brightcove.net, but the runtime execution context depends on how the player is embedded.

The one sentence to teach

Script delivery origin and player runtime context are not the same thing.

Video Tag

The script is delivered from players.brightcove.net, but the player runs inside the current page DOM.

iFrame

The iframe loads a hosted player page, so the player runtime is inside that hosted frame.

Player URL

The hosted player page is opened directly, so the runtime is fully on the hosted player page.

That is why a site-only allowlist usually permits the video tag path but blocks the iframe and hosted player URL paths.

Technical notes and debug checklist

Keep the guidance structural: what was configured, where playback was evaluated, and what result that should produce.

Troubleshooting

Technical notes

What to teach engineers

  • Domain restriction should always be interpreted together with the embed method being tested.
  • A video tag embed is usually the clearest way to verify behavior in the current page context.
  • iframe tests can be misleading when engineers assume the visible host page is always the restriction evaluation target.
  • Testing the hosted player URL helps isolate player-side behavior from integration-side behavior.
  • If results look inconsistent, compare the configured allowlist against the actual runtime context observed by the player.
