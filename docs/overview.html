What the counters mean

  • Manifest: .mpd (DASH) or (heuristic) HLS master .m3u8.
  • Playlist: HLS media .m3u8 listing segments.
  • Init segment: init*.m4s/mp4 or URLs containing initialization (fMP4/CMAF).
  • Segments: .ts / .m4s (excluding init).
  • Poster: .jpg/.png/.webp (may vary due to responsive sizing/CDN optimization).

v4 adds Bytes (NET/ALL) and ABR hints. If you see multiple ABR keys early, the player may be probing multiple renditions.

How Streaming Works (Simple Explanation)
A literacy-friendly view of HLS/DASH loading flow

When a video player loads a streaming video (HLS or DASH), it does not download the entire video at once. Instead, it downloads several small files in stages so playback can start quickly and adapt to network conditions.

1) Poster Image (Optional)

Before playback starts, the player may load a poster image. This is the thumbnail image displayed before the user presses play. Poster is not part of the streaming video, but it improves UX by showing a preview.

  • poster.jpg / poster.png / poster.webp

2) Manifest (Streaming Entry Point)

The first streaming file is the manifest (a table of contents for the stream). It tells the player what qualities (renditions) exist and where each rendition’s playlist is located.

  • master.m3u8 (HLS) / manifest.mpd (DASH)
  • Example renditions: 1080p (high), 720p (mid), 360p (low)

3) Playlist (Segment List)

After choosing a rendition, the player downloads a playlist. This contains a list of small media chunks (segments). For live streams, this playlist may update repeatedly so the player can follow the newest segments.

  • playlist_720p.m3u8 (example)
  • Lists: segment0, segment1, segment2 …

4) Init Segment (For fMP4 / CMAF streams)

Modern streaming often uses fMP4/CMAF. In that case, the player may download an init segment once, before downloading media segments. It contains decoder configuration (codec/track/container metadata).

  • init.mp4 / init.m4s / URLs containing initialization

5) Media Segments (Actual Video/Audio Data)

The actual video/audio arrives as segments (small chunks, often 2–6 seconds each). The player downloads these sequentially, buffering ahead to keep playback smooth. This design enables fast startup and ABR (adaptive bitrate switching).

  • HLS TS: segment0.ts, segment1.ts …
  • HLS/DASH fMP4: segment0.m4s, segment1.m4s …

Why This Matters for Preload

The preload setting controls how many of these files are fetched before the user presses play. That’s why preload affects startup speed, bandwidth usage, and (often) CDN cost.

preload = none → usually Poster only
preload = metadata → Poster + Manifest + Playlist (structure, little/no media)
preload = auto → Poster + Manifest + Playlist + (Init) + first media segment(s)
